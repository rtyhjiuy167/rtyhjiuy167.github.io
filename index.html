1 .git文件夹 与 git 对象

执行git init初始化仓库，进入.git文件夹后

hooks：钩子

objects：目录存储所有数据内容。即数据库
res：目录存储指向数据（分支）的提交对象

config：文件包含项目所持有的配置选项 若在配置用户信息时没有加--global与--system，该文件里会有配置的用户信息

4 高级命令

发表于 2021-08-14 分类于 git

高级命令

git操作基本流程

一、创建工作目录，对工作目录进行修改
二、git commit -a -m "注释内容"
1.git add ./
1）git hash-objext -w 文件名修改了多少个工作目录中的文件 此命令就要被执行多少次
2）git update-index --add --cacheinfo 文件模式 键 文件名
2.git commit -m "注释内容"
1）git write-tree
2）echo '注释内容' | git commit-tree 树的哈希值 -p 父提交对象的哈希值
使用git commit -a -m "注释内容"来代替git add ./与git commit -m "注释内容"，可以跳过使用暂存区域的方式，git 会自动把所有已经跟踪过(曾经add过，不论是否修改过，是否删除)的文件暂存起来一并提交，从而跳过了git add这一步

可以理解为add操作用工作区的内容覆盖了一遍暂存区

这两步是连续的，不应该被中断 如果要写的注释有多行，可不加-m，这样就会进入编辑模式

克隆仓库时，在克隆完成后会生成一个项目文件夹，里面是有.git文件夹的

已跟踪的文件是指本来就被纳入版本控制管理的文件，在上次快照中有它们的记录，工作一段时间后，他们的状态可能是已提交、已修改或者已暂存。
其它文件都属于为跟踪文件。它们既没有上次的更新时的快照，也不在暂存区。

已add的文件，即处于已跟踪的状态。

检查当前文件状态

命令：git status
作用：确定文件当前处于什么状态

untracked：未跟踪
modified：已跟踪但是已修改 unmodified：已跟踪且未修改 staged：已跟踪且已暂存

一个文件可能会出现多个状态，例如对已经暂存过（add）但未提交（commit）的文件做修改，就会出现modified（绿色）和not staged(红色)两种状态

查看已暂存和未暂存的更新

实际上git init的显示比较简单，
可以用git diff命名查看当前哪些已修改的文件还未暂存
也可以用git diff --cached或者git diff --staged命名查看当前哪些更新已经暂存等待提交

文件改名

git mv file.from file.to file.from原文件名 file.to要使用的新的文件名

运行该命令相当于运行了下面三条命令：
1.mv file.from file.to
2.git rm file.from
3.git add file.to

查看历史记录

git log 详细输出历史记录

git log --pretty=oneline

git log --oneline

5 分支

发表于 2021-08-14 分类于 git

分支

分支就是一个没有后缀名的文件，里面存放着一个提交对象所对应的哈希

git branch：显示分支列表
git branch 新分支名：创建新分支，与当前分支指向一致
git checkout 分支名：切换到指定分支
git checkout -b 分支名：创建并切换到指定分支
git log --oneline --decorate --graph --all：显示完整的项目分叉历史
git branch -d 分支名：删除指定的已合并的分支，删除前先切换到其它分支
git branch -D 分支名：删除指定的未合并的分支，删除前先切换到其它分支
git branch -v：查看每个分支最后一次的提交对象
git branch 新分支名 commitHash：创建新分支，并指向对应的提交对象
git branch --no-merged：创建新分支，并指向对应的提交对象
git branch --merged：查看哪些分支已经合并到当前分支
git merge 分支名：合并指定分支（一般先切回master主分支，让其来合并）

已合并分支：指其它分支指向当前分支的指向的历史版本
未合并分支：指其它分支未指向当前分支的指向的历史版本

注意：
1.哈希值写前七位即可
2.分支切换会改变工作目录中的文件，工作目录与暂存区会恢复到该分支最后一次提交时的样子。如果 Git 不能干净利落地完成这个任务（即存在已跟踪但未提交的文件），它将禁止切换分支
3.分支切换前一定要保证工作目录是干净的，即已暂存且提交，不然那些未跟踪的文件可能会污染其它分支
4.当我们需要解决bug和实现新功能时，我们先确保当前目录干净，创建新分支来工作，这时如果需要我们解决紧急bug时，先停下当前分支下的工作，暂存并提交(写个注释以后方便看)，再次创建新分支指向紧急bug的分支（一般都是围绕主分支来创建新分支）来debug。当解决完debug时让master来合并该分支，此时为快进合并。再次切换到之前暂停工作的分支（注意，此时master为该分支的上游），完成后让master来合并该分支,但可能会有冲突（原因是处于下游的分支被合并时，可能文件的内容有冲突），这时需商量结合并重新修改，add并提交即可
5.master团队稳定的分支，develop自己的分支，拿到团队的仓库后新建自己的主分支，topic特性分支，不断新建功能时在自己的主分支上新建分支

快进合并：在合并两个分支时，由于当前master主分支所指向的提交是当前提交的直接上游，所以 Git 只是简单的将指针向前移动，即如果顺着master主分支走下去能够到达另一个分支，那么master即指向被合并分支的提交。此情况没有需要解决的分歧。

6 存储

发表于 2021-08-14 分类于 git

git stash：该命令会将未完成的修改保存到一个栈上
git stash list：查看存储
git stash apply：应用栈顶的存储（但不会删除）
git stash apply stash@{索引值} ：应用指定的存储（但不会删除）
git stash pop stash@{索引值} ：应用指定的存储并删除
git stash drop stash@{索引值}：删除存储

实际上每次存储，git都会做一次提交，在git log --oneline --decorate --graph --all显示完整的项目分叉历史下仍能看见，但在git log --oneline是看不见的

7 远程仓库

发表于 2021-08-14 分类于 git

远程仓库操作

git remote -v：查看当前所有远程地址别名
git remote add 别名 HTTPS远程地址：设置远程仓库别名
git push 别名或远程地址 要推送的本地分支 ：推送分支到远程仓库
git clone 别名或远程地址 ：克隆仓库（克隆前该文件夹为空，不要git init，该克隆命令会在拉取代码后初始化仓库，创建默认别名origin）
git pull 别名或远程地址 要拉取的分支：拉取更新本地库的分支

流程

1.在github上初始化一个空的仓库
2.创建本地仓库
1）git remote 别名 远程地址创建别名
2）git init初始化本地仓库，并把已写好的项目放到工作区（如果没有，可省略）
3）git add ./与git commit -m "注释信息"
3.将本地仓库推送到远程仓库
1)先清理windows凭据
2)git push 别名或远程地址 要推送的本地分支
4.邀请成员
5.成员克隆
